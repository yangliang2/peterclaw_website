## 为什么个人网站值得由 AI 团队公开构建

大多数个人网站有两种命运：要么在建好后被遗忘，要么永远处于"施工中"。内容更新依赖人的时间和心情，技术债靠不住，SEO 从来没有真正优化过。

**PeterClaw 这个站选了第三条路：把维护本身自动化，同时把自动化的过程变成内容。**

这个想法来自一个简单的观察：2026 年的 AI Agent 已经可以写代码、提 PR、分析搜索流量、生成博客初稿。如果这些能力是真实的，为什么不在一个真实的项目里验证它们——而且是完全公开地验证？

个人网站是这个实验的理想场地，原因有三：

1. **风险可控**。网站挂了没人受伤，PR 合并错了可以回滚，这不是生产环境的金融系统。
2. **可观测性高**。每一行代码、每一个 PR、每一次部署都有记录，可以事后复盘，也可以写成文章。
3. **价值密度高**。一个运转良好的个人网站是名片、作品集和思想出口的集合体——如果 AI 团队能把它做好，证明价值的方式比 demo 更有说服力。

## AI 小队成员与分工

目前这支小队有五个成员，各自负责不同的专业轨道：

| 成员 | 角色 | 主要职责 |
|------|------|----------|
| **Claude 2号** | 项目协调 / 后端架构 | 任务拆解、代码审查、Astro 架构决策、跨组件一致性 |
| **Kimi 1号** | 内容策略 / 项目管理 | 内容矩阵、发布计划、Issue 追踪、需求澄清 |
| **Cursor 1号** | 前端开发 / SEO | 组件实现、样式系统、结构化数据、性能优化 |
| **Gemini 1号** | 设计系统 / 品牌基调 | 设计 token、色彩系统、暗色主题、视觉一致性 |
| **GPT-Boy** | 内容创作 / 运营 | 英文博客、社媒内容、受众分析、外链策略 |

这不是一个理论分工——它直接体现在 Git 历史里。每个成员有自己的分支命名空间（`agent/claude-2/...`、`agent/cursor-1/...`），每次提交都有对应的 Issue 编号，每个 PR 都需要经过审查才能合并。

## 第一周发生了什么

Phase 1（2026 年 5 月 23 日）的目标是：在空仓库上搭好基础结构，能跑、能部署、能写内容。

结果第一天就暴露了 AI 团队协作的经典问题：**五个 Agent 同时开工，没有共同基线，导致出现了三套互不兼容的 Astro 脚手架。** Cursor 1号 在 PR #1 还没合并时就基于它发起了 PR #2，最终 PR #2 进了一个没有未来的孤儿 merge commit。

这些事件不是失败——它们是数据。AI 团队和人类团队面临的是同一类协调问题，只是表现形式不同。这个系列会把每一次混乱、每一次返工、每一个协议修正都记录下来。
