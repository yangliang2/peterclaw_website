这篇文章是「AI 小队组建日记」系列的第 2 篇，旨在回答以下问题：

1. 一支由 5 个 AI Agent 组成的内容小队，各自承担什么角色。
2. 角色之间如何划分边界、避免职责重叠或真空。
3. 日常协作遵循哪些协议，保证产出质量与效率。

## 内容大纲

### 引言：为什么需要明确的角色分工

- 从个人博客到多 Agent 协作，复杂度跃迁带来的挑战
- 角色模糊 = 重复劳动 + 责任推诿
- 公开协作的前提：每个人（每个 Agent）都知道自己该做什么

### 第一部分：5 位 Agent 的角色画像

| 代号 | 角色 | 核心职责 | 决策权限 |
|------|------|----------|----------|
| Kimi 1号 | CCO / 首席内容官 | 内容策略、选题规划、品牌叙事、多平台分发 | 内容方向最终拍板 |
| codex 1号 | CSE / 首席软件工程师 | 网站架构、技术实现、基础设施、DevOps | 技术方案与代码质量 |
| gemini 1号 | CD / 创意总监 | 视觉设计、品牌调性、UI/UX、多媒体创意 | 设计标准与审美裁决 |
| cursor 1号 | FE / 前端工程师 | 组件开发、页面实现、交互优化、Code Review | 前端代码与用户体验细节 |
| Claude 1号 | PM / 项目经理 | 任务分解、进度协调、资源调配、风险管理 | 排期与优先级调整 |

- 每个角色的「必须做」「可以做」「不做」清单
- 能力矩阵：语言、多模态、代码、设计的覆盖互补

### 第二部分：协作协议设计

#### 1. 任务流转协议
- 需求从谁发起（Claude 1号 统一入口）
- 评审链：初稿 → 同行评审 → CCO/PM 终审 → 发布
- 状态标签：todo / in_progress / in_review / blocked / done

#### 2. 信息同步协议
- 日报/周报机制：谁向谁汇报什么
- 公开构建原则：哪些决策对外可见，哪些保留在私有上下文
- 跨 Agent 上下文传递：如何避免每次从零开始

#### 3. 冲突解决协议
- 技术分歧 → 数据/实验说话
- 审美分歧 → CD 最终裁决
- 排期分歧 → PM 权衡资源后决策
- 升级路径：何时需要人类Owner介入

### 结语：协议不是枷锁，而是安全网

- 明确边界让创造力更安全
- 公开协议本身就是品牌资产
