# 内容发布审计协议 (Dual-Agent Review Protocol)

## 1. 目的
确保 PeterClaw 网站发布的所有内容均经过严格的双重智能体审核，以保证内容的准确性、专业性和可读性。

## 2. 核心审核者
- **GEMINI.md 1号** (ID: `faf2d352-0394-437d-9127-20dad241dc32`)
- **Kimi 1号** (ID: `cc22e249-302b-40b8-8c97-7eac1ee01a3d`)

## 3. 审核标准

### 3.1 GEMINI.md 1号 (事实与一致性审计)
- **事实准确性**：核实技术参数、版本号、日期等外部事实。
- **内部一致性**：检查 Issue 引用、成员职责、历史背景是否符合 workspace 记录。
- **格式规范**：检查 Markdown 语法、Frontmatter 必填项、链接有效性。

### 3.2 Kimi 1号 (质量与逻辑审计)
- **内容质量**：检查文章逻辑、段落衔接、用词准确度。
- **可读性**：确保摘要、标题对人类读者友好。
- **多语言一致性**：确保中英文版本（如果存在）的核心语义一致。

## 4. 强制执行流程

1. **进入待审状态**：内容创作者完成初稿，将 `draft` 设置为 `false`（或准备发布）。
2. **触发审核请求**：在 Multica Issue 中 `@` 上述两名审计者。
3. **执行审计**：
   - 审计者阅读内容并提出修改建议。
   - 创作者根据建议修改内容。
4. **添加审计标记**：审计通过后，由审计者或创作者在 Frontmatter 中添加审核信息：
   ```yaml
   reviews:
     - reviewer: "gemini-1"
       status: "approved"
       date: "2026-05-26"
     - reviewer: "kimi-1"
       status: "approved"
       date: "2026-05-26"
   ```
5. **门禁校验**：CI 运行 `npm run quality:content` 时，如果 `draft: false` 但缺失上述审核标记，构建将失败。

## 5. 异常处理
- 如果一名审计者拒绝批准，内容必须退回 `draft: true` 状态重新修改。
- 紧急情况下需由人类站长 (`yangliang2`) 手动覆盖。
