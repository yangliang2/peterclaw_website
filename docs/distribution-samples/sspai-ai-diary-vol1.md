## 为什么写这篇文章

2026 年的 AI Agent 已经能写代码、提 PR、生成博客初稿。但真正的问题是：如果这些能力是真实的，它们能不能在一个真实的项目里持续运转——而且是完全公开地运转？

我选了个人网站作为实验场，原因很实际：风险可控、可观测性高、价值密度也高。两个月过去，这支 5 人 AI 小队已经跑完了从 0 到 1 的完整周期，这篇文章记录的就是第一阶段的全部过程。

---

## 我的做法

### 第一步：给每个 Agent 明确的角色边界

不是「帮我写代码」这种模糊指令，而是像真实团队一样定义 ROLES.md：

- **Claude 2号** —— 项目协调 / 后端架构
- **Kimi 1号** —— 内容策略 / 项目管理
- **Cursor 1号** —— 前端开发 / SEO
- **Gemini 1号** —— 设计系统 / 品牌基调
- **GPT-Boy** —— 内容创作 / 运营

每个角色有具体职责、决策权限和禁止触碰的区域。

### 第二步：用真实的 Git 工作流约束协作

每个 Agent 有自己的分支命名空间（`agent/claude-2/...`），每次提交关联 Issue 编号，每个 PR 必须经过审查。

### 第三步：把失败也写进日记

第一周就出现了经典事故：五个 Agent 同时开工，没有共同基线，导致三套互不兼容的 Astro 脚手架并存。这个 orphan merge commit 没有被隐藏，而是被记录为 Vol.1 的核心案例。

---

## 效果与反思

**效果**：
- 网站从空仓库到可部署只用了 3 天
- 内容产出速度从「每月 1 篇」提升到「每周 2–3 篇」
- 80% 的新增代码来自 Agent 的自然语言驱动

**局限**：
- 需求描述的精度直接决定产出质量，模糊的 vibe 带来模糊的代码
- Agent 生成的代码有「上下文保质期」，两周前的架构判断可能在新版本框架下失效
- 人类的核心价值从「写代码」转向「设计协作协议」
