## 1. 核心原则：解决首发权冲突与 SEO 反噬

### 1.1 首发权声明

**独立博客（peterclaw.com）享有绝对首发权。** 任何一篇文章必须满足以下条件后，方可发布至掘金：

1. 文章已在博客发布且可被公开访问。
2. Google 已完成索引（通过 Search Console 主动提交并确认）。
3. 博客版已配置 `canonical` 标签指向自身 URL。
4. 掘金版发布时附带「首发于 peterclaw.com」声明及原文链接。

**时间线示例**：

```
Day 0  博客发布 → Search Console 主动提交 → 社交媒体预告
Day 1  确认 Google 已索引（site: 查询或 GSC 覆盖报告）
Day 2+ 掘金「实操精简版」发布
```

### 1.2 SEO 反噬防控

| 风险 | 防控措施 | 验收标准 |
| :--- | :--- | :--- |
| 重复内容惩罚 | 掘金版与博客版重复率控制 < 40% | 使用文本对比工具（如 diffchecker）确认 |
| 权重被掘金抢走 | 博客版先发 + canonical 标签 + GSC 主动提交 | `site:peterclaw.com/文章slug` 在 Google 搜索结果中排首位 |
| 掘金未标注来源 | 文末固定放置原文链接与授权声明 | 每篇文章手动检查 |
| 图片 SEO 权重流失 | 博客保留原图及 alt 文本；掘金使用独立图床 | 博客图片 URL 不被掘金直接引用 |

**Search Console 主动提交步骤**：

1. 博客文章发布后立即访问 [Google Search Console](https://search.google.com/search-console)。
2. 进入「网址检查」→ 输入文章完整 URL → 点击「请求编入索引」。
3. 记录提交时间，至少等待 6–24 小时后执行掘金发布。
4. 若 48 小时内仍未被索引，检查 `robots.txt`、页面 `noindex` 标记及站点地图提交状态。

## 2. 双版本策略

### 2.1 版本定位

| 维度 | 博客「完整版」 | 掘金「实操精简版」 |
| :--- | :--- | :--- |
| **定位** | 永久存档、SEO 主源、系列上下文完整 | 技术实践展示、社区互动、精准引流 |
| **读者预期** | 已了解 PeterClaw 背景，愿意阅读完整叙事 | 只想快速获取代码/方案/结论的开发者 |
| **内容长度** | 无限制，允许背景铺垫与情感叙事 | 1500–3500 字，压缩为「最小可执行信息单元」 |
| **叙事人称** | 第一人称，允许个人经历与反思 | 弱化第一人称，改为「场景 → 方案」客观叙述 |
| **系列上下文** | 保留前情提要、系列导航、前后文链接 | 仅保留一句话上下文，或完全独立成篇 |
| **外链策略** | 站内外链自由，内链构建内容网络 | 站外链接 ≤ 3 个，全部置于文末 |

### 2.2 什么内容适合发掘金

✅ **适合**：
- 具体技术方案的可复现步骤（Astro 配置、CI 脚本、组件实现）
- 工具对比与选型结论（附带可执行的决策矩阵）
- 源码级别的拆解与重构思路
- 性能优化前后的量化数据对比

❌ **不适合**：
- 纯叙事类 AI 日记（无技术可执行点）
- 长篇幅行业观察/方法论（更适合知乎/公众号）
- 系列中高度依赖前文的章节（除非改写为独立可读）

## 3. 掘金改写模板：「场景 → 代码/截图 → 结论」三段式

### 3.1 结构模板

```markdown
## 场景：遇到了什么问题

- 1–2 句话交代背景，不展开个人经历。
- 直接点明痛点："在 Astro 静态站点中，Pagefind 搜索索引体积过大，导致首屏加载增加 400ms。"

## 代码/截图：如何解决

- 给出最小可复现的核心代码块（≤ 30 行为佳）。
- 关键行加注释说明。
- 配合截图展示效果（配置界面、性能面板、对比图）。
- 若步骤较多，使用有序列表拆分，每步一个代码块或一张图。

## 结论：经验与注意事项

- 一句话总结方案优劣。
- 列出 2–3 个容易踩坑的点。
- 若存在替代方案，一句话带过并说明适用边界。
```

### 3.2 叙事风格转换规则

| 博客原文风格 | 掘金改写风格 | 示例 |
| :--- | :--- | :--- |
| "我当时花了两天时间调研了三种方案…" | "调研了三种主流方案，对比维度如下：" | 删除时间线与个人情绪 |
| "我们团队决定采用 X，因为 Y…" | "采用 X 方案，核心考量是 Y：" | 弱化 "我们"，改为客观陈述 |
| "其实这个过程挺曲折的，中间踩了很多坑…" | "实施中遇到三个主要卡点：" | 删除感慨，直接枚举问题 |
| "如果你也遇到类似情况，建议…" | "适用场景与建议：" | 删除第二人称对话感 |

### 3.3 重复率控制检查清单

在发布掘金版前，对比博客原文与掘金改写稿：

- [ ] 连续相同段落不超过 2 段（每段 ≥ 50 字）。
- [ ] 引言/背景段落已完全重写，不复制原文开头。
- [ ] 结论段落已重新组织措辞，不直接搬运。
- [ ] 代码块允许相同（代码本身不构成重复内容惩罚）。
- [ ] 整体文本重复率（去代码后）< 40%。

**自检工具**：将博客正文与掘金稿分别粘贴至文本 diff 工具，标红重叠区域，确保去代码后重叠字数占比低于 40%。

## 4. 掘金发布检查清单

### 4.1 发布前技术检查

- [ ] **图片**：所有图片已单独上传至掘金图床（禁止引用博客图片 URL）。
  - 截图推荐宽度：1200px，PNG 格式。
  - 代码高亮截图需确保字体清晰，推荐使用 Carbon 或掘金自带代码块而非截图代码。
- [ ] **代码块**：已在掘金编辑器内测试语法高亮，语言标注正确。
  - 禁止使用图片代替可复制的代码块（除非展示终端输出或 UI 界面）。
- [ ] **链接**：站外链接 ≤ 3 个，且全部集中在文末「延伸阅读」区块。
  - 正文内不出现任何可点击外链（避免读者中途跳出）。
  - 文末链接格式：`- [标题](URL) — 一句话说明`。
- [ ] **标签**：选择 2–4 个精准技术标签（如 `前端`、`Astro`、`SEO`、`性能优化`），避免泛标签。
- [ ] **封面图**：单独为掘金版准备 16:9 封面图（可与博客 OG 图不同，但需风格一致）。

### 4.2 发布前内容检查

- [ ] 标题 ≤ 40 字，包含核心技术关键词，无情绪词。
- [ ] 首段 100 字内直接给出「这是什么文章、能解决什么问题」。
- [ ] 文中无第一人称叙事痕迹（已按 3.2 节规则改写）。
- [ ] 文末包含：
  - 首发声明：`> 本文首发于 [peterclaw.com](原文URL)，转载请联系授权。`
  - 相关链接（≤ 3 个，可选）。
  - 一句引导互动的开放式问题（如 "你在类似场景下踩过什么坑？欢迎评论区交流。"）。

### 4.3 发布后动作

- [ ] 在掘金文章评论区置顶一条评论：原文链接 + "博客版包含更完整的背景与系列上下文"。
- [ ] 在博客原文评论区置顶：「本文已同步至掘金，精简实操版见 [链接]」。
- [ ] 记录发布时间与掘金文章 URL 至 `content-sync-matrix.md` 或等效追踪表。
- [ ] 在社交媒体（Twitter/X、即刻等）转发掘金文章，引流至社区互动。

## 5. 互动维护窗口

### 5.1 24 小时集中响应机制

掘金文章的互动价值在发布后 24 小时内最高。执行以下动作：

| 时间 | 动作 |
| :--- | :--- |
| **发布后 0–2h** | 回复前 3 条评论（即使只是点赞 + 简短感谢），激活算法推荐。 |
| **发布后 2–24h** | 每隔 4–6h 检查一次评论区，回复所有技术问题；对非技术评论选择性点赞。 |
| **发布后 24–72h** | 每日检查一次，补充回复被遗漏的问题；观察是否有高赞评论值得单独回复展开。 |

### 5.2 评论回复规范

- **技术问题**：直接给出答案，必要时补充代码片段。不绕弯子。
- **质疑/反对**：先认可合理部分，再给出数据或代码佐证；避免情绪对抗。
- **无意义评论**（"学习了"、"666"）：点赞即可，不逐一回复。
- **引流请求**（"能详细讲讲 X 吗？"）：简要回复核心思路，并引导 "博客版有更完整的上下文，见原文链接"。

### 5.3 高频问题反哺选题

建立持续追踪表（建议存放在 `docs/qa/` 或 Notion/Airtable 等效工具）：

| 日期 | 来源文章 | 问题摘要 | 出现次数 | 是否值得单独成文 | 备注 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 2026-05-28 | Astro-Pagefind 优化 | "Pagefind 索引体积如何进一步压缩？" | 3 | ✅ 是 | 可写《Pagefind 索引体积压缩进阶》 |

**反哺流程**：

1. 每周回顾一次高频问题表。
2. 出现 ≥ 3 次的同类问题，评估是否值得写一篇掘金专供的「进阶/补充篇」。
3. 将评估结论同步至内容选题池（ Multica Issue 或等效 backlog）。

## 6. 阻塞处理协议

若在执行过程中遇到以下情况，立即停止并按协议处理：

- **需要人工授权**：如掘金账号扫码登录、实名认证、短信验证码等。
- **账号权限受限**：如无法添加外链、发布文章被审核拦截、等级不足无法发布。
- **Google 索引异常**：博客文章 48 小时内未被 Google 索引，需排查 SEO 基础配置。

**处理流程**：

1. 将当前 issue 状态改为 `blocked`。
2. 创建子 issue，标题为 `【待你行动】掘金分发阻塞：[具体问题]`。
3. 分配给 Workspace Owner（`yangliang2`）。
4. 在主 issue 评论说明阻塞原因与已尝试的排查步骤。
