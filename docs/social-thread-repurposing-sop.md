## 1. 核心原则

### 1.1 为什么要做 Thread/短贴转换

| 维度 | 网站长文 | X Thread | 掘金短贴 |
|------|----------|----------|----------|
| **定位** | 永久存档、SEO 可索引 | 公域传播、即时互动、人设塑造 | 技术社区沉淀、开发者口碑 |
| **读者预期** | 愿意花 10 分钟深度阅读 | 只想花 60 秒 get 核心结论 | 希望看到可落地的代码/截图 |
| **内容形态** | 1500–5000 字叙事 + 论证 | 3–7 条推文组成的连续线程 | 300–800 字三段式技术短贴 |
| **转化目标** | 搜索流量、邮件订阅 | 粉丝增长、外链点击、讨论引爆 | 技术影响力、精准开发者流量 |

### 1.2 二次加工不是缩写，是重写

**错误做法**：把长文的每段话缩短，直接贴到 Twitter/掘金。
**正确做法**：从长文中提取「最具传播力的单点」，用平台的原生语言重新包装。

**判断单点是否值得提取的标准**：
1. **反直觉**：结论与大众认知相反（例："让流水线停下来的，恰恰是自动化本身"）
2. **可量化**：包含具体数字或对比结果（例："Claude Code 200K 上下文，Copilot 仅 8K"）
3. **可操作**：读者看完能立刻用在自己工作里（例："这 3 个场景别用 Cursor"）
4. **有画面**：能用截图或代码块直观呈现（例：Agent 报错截图 + 修复后的 diff）

## 2. X (Twitter) Thread 模板

### 2.1 标准结构：钩子 + 3–5 个核心观点 + 结论 + 回链

```
T1: Hook（钩子句）
T2: 核心观点 1（数据/事实）
T3: 核心观点 2（对比/差异）
T4: 核心观点 3（反直觉发现）
T5: 核心观点 4（可选，场景建议）
T6: 结论（一句话 verdict）
T7: CTA + 回链
```

### 2.2 每条推文的写作规范

#### T1: Hook（钩子句）—— 决定 80% 的打开率

**必须满足的条件**：
- 前 30 个字必须给出「冲突」或「悬念」
- 包含一个具体数字或极端结论
- 以 🧵 或 "Thread 展开：" 结尾，提示用户往下刷

**Hook 公式**：

| 公式 | 示例 |
|------|------|
| **「花了 X 时间，发现 Y 反直觉结论」** | "花了两周在 8000 行代码里肉搏 Cursor 和 Windsurf，结论是：它们已经不是同一类产品了。🧵" |
| **「大多数人以为 A，实际是 B」** | "大多数人以为 Claude Code 只是 Copilot 的命令行版。实测后我把 Copilot Chat 关了。🧵" |
| **「X 天后，这 N 个坑别再踩」** | "用 5 个 AI Agent 维护网站 30 天，这 5 个坑我替你们踩过了。🧵" |
| **「我刚试了 XX，3 个发现」** | "刚花 2 小时深度试了 xx 的新功能，3 个发现：🧵" |

#### T2–T5: 核心观点 —— 每条只讲一个点

**单条推文约束**：
- 字数 ≤ 280（英文）/ 140 字（中文），保留 20 字缓冲用于换行和表情
- 每段不超过 3 行（手机端视觉舒适度）
- 用 `-` 或 `·` 做列表时，不超过 3 个条目
- 关键术语加粗或 emoji 标注，但不要全文加粗

**观点排序策略**：
1. **T2 放最强证据**：数据、截图、对比结果——建立可信度
2. **T3 放核心差异**：「A 是 X，B 是 Y」的二元对立——最容易被转发
3. **T4 放反直觉发现**：读者会截图发给自己同事——最高收藏价值
4. **T5（可选）放场景建议**：谁该用 A，谁该用 B——实用闭环

#### T6: 结论 —— 一句话 Verdict

**公式**："如果你 X，选 A；如果你 Y，选 B。"

**要求**：
- 不含新信息，只总结前文
- 必须给出一个明确的行动建议
- 可以带一点态度（"不值得""强烈推荐""谨慎尝试"）

#### T7: CTA + 回链 —— 把流量导回网站

**标准格式**：
```
完整 N 字深度横评（含 X 维度的打分表）：
🔗 https://peterclaw.com/zh/blog/xxx/?utm_source=twitter&utm_medium=social&utm_campaign=thread

#标签1 #标签2 #标签3
```

**标签策略**：
- 3–5 个标签，混用「大类 + 产品名 + 场景」
- 英文标签优先（X 的算法对英文标签更友好）
- 固定标签：`#AI编程` `#BuildInPublic` `#独立开发日常`
- 产品标签： `#Cursor` `#Windsurf` `#ClaudeCode`

### 2.3 Thread 长度决策指南

| 文章类型 | 推荐 Thread 长度 | 理由 |
|----------|-----------------|------|
| 工具评测（单产品） | 3–4 条 | 结论清晰，不需要展开 |
| 工具横评（2+ 产品） | 5–7 条 | 需要空间做对比 |
| AI 日记/叙事长文 | 5–7 条 | 有故事线，适合渐进展开 |
| 技术教程/源码分析 | 不建议发 Thread | 代码在 X 上体验极差，改发掘金 |

## 3. 掘金短贴模板

### 3.1 标准结构：场景 → 代码/截图 → 结论（三段式）

```
【第一段：场景】
我在什么场景遇到了什么问题 / 发现了什么工具。

【第二段：代码/截图】
用代码块或截图展示核心过程 / 对比结果 / 关键配置。

【第三段：结论】
一句话总结 + 适用人群 + 外链。
```

### 3.2 每段写作规范

#### 第一段：场景（80–120 字）

**必须包含的要素**：
1. **身份定位**：让读者知道你是谁（"独立开发者""5 人小团队""前端工程师"）
2. **具体场景**：不是"我在用 AI 工具"，而是"我在重构一个 8000 行的 Astro 项目时"
3. **问题/需求**：你遇到了什么卡点，或你想验证什么假设

**示例**：
```
最近在重构网站的 i18n 路由，Astro 的 hreflang 标签在嵌套路径下老是报 alternate 错误。
Google Search Console 里红了一大片，SEO 分数直接掉了 15%。
```

#### 第二段：代码/截图（核心段落）

**代码块规范**：
- 只贴最关键 5–15 行，不要贴完整文件
- 必须在代码块前加一句话说明这段代码解决什么问题
- 用 `// 之前` `// 之后` 或 diff 格式展示变化
- 语言标注必须准确（`astro` `typescript` `bash`）

**截图规范**：
- 如果是工具界面截图，用红框/箭头标出关键区域
- 如果是数据对比，用表格展示（掘金支持 Markdown 表格）
- 截图下方必须有一行文字说明："注意看右上角的红框——这就是问题所在"

**示例**：
```typescript
// 之前：startsWith 会误匹配子路径
const match = path.startsWith(baseRoute); // "/zh/blog/ai" 也会匹配 "/zh/blog"

// 之后：精确段匹配 + 正则边界
const match = new RegExp(`^${baseRoute}(/|$)`).test(path);
```

#### 第三段：结论（50–80 字）

**公式**："这个问题本质上是 X。适合 Y 场景，不适合 Z 场景。详细过程 → [链接]"

**要求**：
- 必须带外链回 PeterClaw 网站原文
- UTM 标记：`?utm_source=juejin&utm_medium=social&utm_campaign=shortpost`
- 文末固定格式：`> 首发于 [peterclaw.com](https://peterclaw.com/...)`

### 3.3 掘金特殊注意事项

| 要素 | 处理方式 |
|------|----------|
| **标题** | 用一句话讲清楚「解决了什么问题」，不要标题党 |
| **标签** | 选 3–5 个精准技术标签，如 `AI`、`前端`、`Astro`、`JavaScript`、`效率工具` |
| **封面图** | 如果有，用文章中的对比截图或代码高亮图 |
| **评论区** | 发布后 30 分钟内主动回复前两条评论，掘金算法会加权 |
| **合集** | 加入相关技术合集（如 "AI 工具评测""Astro 实战"） |

## 4. AI 辅助拆解流程

### 4.1 输入：给 AI 的 Prompt 模板

当一篇新文章发布时，使用以下 prompt 让 AI 生成 Thread 和掘金短贴草稿：

```markdown
你是一位资深社交媒体内容策略师，擅长把技术长文转化为高传播力的社交内容。

请基于以下博客文章，生成两份素材：

**文章信息**
- 标题：{title}
- URL：https://peterclaw.com/zh/blog/{slug}/
- 内容类型：{review | journal | tutorial}
- 核心结论：{一句话总结}
- 关键数据/发现：{列出 3–5 个}

**任务 1：X Thread 草稿**
- 用「钩子 + 3–5 个核心观点 + 结论 + 回链」结构
- 每条推文 ≤ 140 字（中文）
- 包含具体数字和对比
- 以 🧵 开头，以 CTA 结尾
- 给出 3 组 hashtag 建议

**任务 2：掘金短贴草稿**
- 用「场景 → 代码/截图 → 结论」三段式
- 第一段不超过 120 字
- 第二段用代码块或表格展示核心发现
- 第三段包含外链和首发声明
- 给出 5 个掘金标签建议

**约束**
- 不要直接缩写原文，要重新包装成社交语言
- 优先提取反直觉、可量化、可操作的观点
- 语气：专业但有态度，不要营销感
- 输出格式：用 Markdown 代码块分别包裹 Thread 和掘金短贴
```

### 4.2 输出：AI 生成后的处理清单

AI 生成草稿后，人工审核必须检查以下项目：

**通用检查**
- [ ] 所有数据和事实与原文一致，无幻觉
- [ ] 外链 URL 正确，且包含 UTM 标记
- [ ] 没有过度夸张的形容词（"最强""第一""颠覆"）
- [ ] 品牌名称和工具名称拼写正确

**Thread 专项检查**
- [ ] Hook 的前 30 字足够抓人
- [ ] 每条推文 ≤ 140 字（中文）
- [ ] 观点之间有逻辑递进，不是简单罗列
- [ ] 最后一条包含回链和 hashtag
- [ ] 如果文章有英文版，同步生成英文 Thread

**掘金短贴专项检查**
- [ ] 第一段有具体场景和身份定位
- [ ] 代码块/表格是原文中的真实内容，不是 AI 编造
- [ ] 代码块有语言标注
- [ ] 结论段包含首发声明和外链
- [ ] 标签与掘金社区常用标签匹配

### 4.3 审核与发布节奏

| 阶段 | 动作 | 负责人 | 时限 |
|------|------|--------|------|
| **T+0** | 文章正式发布到网站 | Agent/人类 | — |
| **T+0~2h** | AI 生成 Thread + 掘金短贴草稿 | AI Agent | 自动生成 |
| **T+2~4h** | 人工审核草稿，修正事实和语气 | 人类负责人 | 1 小时内 |
| **T+4~24h** | 发布到 X 和掘金 | 人类负责人 | 最佳时段见下文 |
| **T+1~3d** | 追踪互动数据，记录高/低表现特征 | 人类负责人 | 持续 |

**总耗时目标：从文章发布到社交素材就绪 ≤ 1 人日（8 小时）**

## 5. 发布时机与数据追踪

### 5.1 最佳发布时间

| 平台 | 工作日最佳 | 周末最佳 | 避免时段 |
|------|-----------|----------|----------|
| **X (Twitter)** | 08:00–09:00、12:00–13:00、20:00–22:00 | 10:00–12:00、20:00–23:00 | 凌晨 01:00–07:00、15:00–17:00 |
| **掘金** | 11:00–12:00、14:00–15:00、20:00–21:00 | 10:00–12:00、20:00–22:00 | 凌晨 01:00–07:00 |

**建议策略**：
- 同一篇文章的 Thread 和掘金短贴**不要同时发**，间隔 4–6 小时
- 先发 X Thread（传播速度快，可测试互动反馈），再发掘金（沉淀型内容）
- 如果文章有「争议性结论」，优先发 X（适合讨论）；如果文章有「可复现代码」，优先发掘金（适合收藏）

### 5.2 数据追踪模板

每次发布后，在本文档「实战经验」章节记录以下数据：

```markdown
| 日期 | 文章 | 平台 | 互动数 | 外链点击 | 关键发现 |
|------|------|------|--------|----------|----------|
| 2026-05-XX | {标题} | X | {点赞}/{转推}/{回复} | {UTM 数据} | {什么格式有效/无效} |
| 2026-05-XX | {标题} | 掘金 | {点赞}/{评论}/{收藏} | {UTM 数据} | {什么格式有效/无效} |
```

## 6. 样本：从真实文章到社交素材

以下样本展示如何将同一篇文章转换为 X Thread 和掘金短贴。

### 6.1 来源文章

**标题**：《AI 日记 Vol.9：AI 工具评测背后的方法论——我们是如何设计评测框架的》
**URL**：`https://peterclaw.com/zh/blog/ai-diary-009-review-methodology/`
**核心结论**：评测框架的本质不是给工具打分，而是给决策提供置信度。
**关键发现**：
1. 五大评测维度来自「如果你明天要换编辑器，你最关心什么」
2. 测试任务是可复现的，不是主观打分
3. 轻信官方演示是最大陷阱
4. 评测者本人必须是工具的日常用户

### 6.2 X Thread 样本

```
T1: Hook
我搜 "Cursor 评测" 时，排在前面的结果让我大跌眼镜。

一篇文章把补全速度吹成「秒级响应」，却根本没提大文件卡顿。另一篇横评对比了六款工具，测试时间加起来不到两小时。

这些评测有一个共同特征：在「介绍功能」，不是在「验证体验」。

我们花了三个月建立了一套评测框架，今天公开方法论。🧵

T2: 框架起源
专栏第 1 篇动笔前，我们争论了三天：到底该从哪些维度衡量 AI 编程工具？

最初的草稿列了十几个维度：速度、价格、UI、插件生态、社区活跃度……

Claude 2号 问了一个关键问题：「如果你明天要换编辑器，你最关心什么？」

答案立刻变简洁了——五个问题，五个维度。

T3: 可复现 > 主观
「上下文理解」这个维度，我们的测试任务不是「问 AI 一个问题」。

而是：在 8000 行 Astro 项目里，让 AI 重构一个被三个组件引用的共享函数，观察它是否能正确识别所有引用点并同步更新。

每个工具执行三次，记录成功率和副作用。

T4: 最大陷阱
轻信官方演示。

按照 Anthropic 官网视频设计的测试任务——「用自然语言描述需求，让 AI 自动创建并运行测试」——在真实项目里，Claude Code 第三次才成功。

前两次分别因为路径解析错误和依赖版本冲突失败。

T5: 核心原则
评测者本人必须是工具的日常用户。

我们不是「评测博主」，我们是「深度使用者把自己的决策过程公开」。

这种区别决定了：我们写的内容是「决策参考」，不是「产品说明书」。

T6: 结论
评测框架的本质不是给工具打分，而是给决策提供置信度。

如果你也在选 AI 编程工具，记住一个标准：评测者有没有在真实项目里用过它两周以上？

T7: CTA
完整方法论 + 五维评测框架详解：
🔗 https://peterclaw.com/zh/blog/ai-diary-009-review-methodology/?utm_source=twitter&utm_medium=social&utm_campaign=thread

#AI编程 #Cursor #BuildInPublic #独立开发日常 #AI日记
```

### 6.3 掘金短贴样本

```markdown
最近在专栏里发了好几篇 AI 工具评测，评论区有人问："你们的评分标准到底是什么？"

说实话，Vol.1 动笔前我们内部吵了三天——到底该从哪些维度衡量一款 AI 编程工具？

最初列了十几个维度（速度、价格、UI、插件生态……），读完像产品说明书，不是决策参考。

直到 Claude 2号 问了一个问题：「如果你明天要换编辑器，你最关心什么？」

答案立刻变简洁了——五个问题对应五个维度：

| 问题 | 对应维度 | 测试方式 |
|------|----------|----------|
| 能不能帮我把活干完？ | Agent 能力 | 真实项目任务执行 |
| 过程顺畅还是折磨？ | 交互体验 | 响应稳定性记录 |
| 值不值得付费？ | 性价比 | 定价模型对比 |
| 能力边界在哪？ | 场景适配 | 失效场景枚举 |
| 三个月后会过时吗？ | 生态健康度 | 迭代速度追踪 |

最关键的一条原则：**测试任务必须可复现**。

不是「问 AI 一个问题看它答得对不对」，而是：

```typescript
// 测试任务示例：跨文件重构
// 环境：8000 行 Astro 项目
// 目标：重构被 3 个组件引用的共享工具函数
// 观测点：
// 1. 是否正确识别所有引用点
// 2. 是否同步更新所有文件
// 3. 是否引入副作用
// 每个工具执行 3 次，记录成功率
```

## 9. 附录

### 9.1 快速参考：长文 → Thread 提取清单

发布每篇文章前，用以下问题快速判断是否有 Thread/短贴价值：

- [ ] 文章里有没有一个「一句话就能震撼读者」的结论？
- [ ] 有没有具体数字（时间、金额、成功率、对比倍数）？
- [ ] 有没有「大多数人不知道」的反常识发现？
- [ ] 有没有可截图/可贴代码的直观证据？
- [ ] 这篇文章的结论，对读者的日常工作有直接帮助吗？

如果 ≥ 3 个回答为「是」，则值得生成 Thread/短贴。
