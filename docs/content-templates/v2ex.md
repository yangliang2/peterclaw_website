# V2EX 独立开发者节点帖子模板
# 使用说明：
# 1. 这不是完整转载，而是「分享 + 钩子」
# 2. 字数控制在 150–400 字
# 3. 必须包含原文链接
# 4. 发布到 creators 节点
---

## 标题格式（选一种）

- 分享型：`分享我刚写的关于 {{主题}} 的实践记录`
- 问题型：`{{结论}}，但我遇到了 {{问题}}，大家怎么看？`
- 数据型：`用 {{方案}} 做了 {{成果}}，分享下过程和踩的坑`

---

## 正文模板

{{一句话说清文章价值：我在实践中做了什么事，得到了什么结果。}}

文章里记录了这几个关键点：

1. {{核心观点 1}}
2. {{核心观点 2}}
3. {{核心观点 3（可选）}}

{{可选：一个具体的数字、截图或代码片段，增加可信度。}}

完整记录在这里：{{原文绝对链接，如 https://peterclaw.com/zh/blog/xxx/}}

欢迎讨论：
- {{开放问题 1：与独立开发者切身相关}}
- {{开放问题 2（可选）}}

---

## 示例

> **标题**：分享我用 5 个 AI Agent 公开构建个人网站的实践记录
>
> **正文**：
>
> 过去两个月我把个人网站的维护完全交给了一支 AI 小队（5 个 Agent 分工协作），并公开记录了所有过程——包括失败的部分。
>
> 文章里记录了这几个关键点：
>
> 1. 多 Agent 并行开发时如何避免代码冲突和孤儿分支
> 2. 用自然语言驱动代码生成的真实边界（不是银弹）
> 3. 自动化 CI 如何替代人工 code review 的部分工作
>
> 第一周我们就踩了一个经典坑：五个 Agent 同时开工，没有共同基线，导致出现了三套互不兼容的 Astro 脚手架。
>
> 完整记录在这里：https://peterclaw.com/zh/blog/ai-squad-launch-diary/
>
> 欢迎讨论：
> - 如果你也在用 AI 辅助开发，是怎么解决多 Agent 协调问题的？
> - 独立开发者有必要把个人网站的维护过程公开化吗？

---

## 发布 checklist

- [ ] 标题像帖子，不像文章
- [ ] 字数在 150–400 字之间
- [ ] 包含原文绝对链接（非短链接）
- [ ] 至少提出 1 个开放讨论问题
- [ ] 语气像聊天，不是「小编推送」
- [ ] 发布到 `creators` 节点
