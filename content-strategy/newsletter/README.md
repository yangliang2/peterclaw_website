## 一、Newsletter 定位文档

### 1.1 名称与标语

| 项目 | 内容 |
|------|------|
| **名称** | PeterClaw Newsletter |
| **中文副标题** | AI 小队构建日志 |
| **英文副标题** | Build Logs from an AI-Native Team |
| **核心承诺** | 每两周交付一份可操作的 AI 实践洞察，来自一个真实运转的 AI 团队 |

### 1.2 读者画像（Reader Persona）

**主要受众：「好奇的实践者」**

| 维度 | 描述 |
|------|------|
| **身份** | 独立开发者、技术写作者、产品经理、AI 早期采用者 |
| **年龄层** | 25–40 岁 |
| **技术水平** | 能独立写代码或至少理解技术概念，关注效率工具 |
| **痛点** | 信息过载但缺乏可验证的实践案例；想引入 AI 到工作流但不知从何开始 |
| **阅读习惯** | 在邮件客户端或 Buttondown 阅读器里快速浏览，遇到有价值的链接会收藏或转发 |
| **语言偏好** | 中文为主（70%），英文为辅（30%） |

**次要受众：「观察者」**

- 对 AI 团队运作方式感兴趣的研究者、创业者
- 关注「公开构建」(Build in Public) 理念的内容消费者

### 1.3 内容主题（Content Pillars）

Newsletter 内容围绕三大支柱展开，每期至少覆盖其中 1–2 个：

| 支柱 | 占比 | 内容方向 | 示例 |
|------|------|----------|------|
| **🛠️ AI 工具实战** | 40% | 真实使用经验、评测精简版、工作流组合 | Cursor/Windsurf 深度使用 2 周后的真实结论 |
| **🚀 独立开发与构建** | 35% | 从想法到上线的完整路径、技术选型、踩坑记录 | 用 Astro + Vercel 搭建内容网站的完整成本清单 |
| **🧠 知识管理与内容生产** | 25% | 个人知识系统、内容发布流水线、信息筛选策略 | 我是如何用 AI Agent 管理 200+ 条内容灵感的 |

### 1.4 差异化价值（Why Subscribe）

与市面上其他 AI/技术 Newsletter 的区别：

| 维度 | 主流 AI Newsletter | PeterClaw Newsletter |
|------|-------------------|----------------------|
| **信息来源** | 编译海外新闻、产品更新 | 来自真实项目的 first-hand 经验 |
| **验证程度** | 多为单次试用或新闻稿改写 | 经过团队多角色协作、多轮 review 的深度实践 |
| **叙事视角** | 第三方评论者 | 第一人称参与者，公开整个决策与失败过程 |
| **可行动性** | 资讯聚合，看完即走 | 每期附「本周可尝试」行动清单，直接可用 |
| **社区感** | 单向广播 | 读者反馈直接影响下期选题，形成共建关系 |

**一句话定位**：
> 这不是一份「AI 新闻摘要」，而是一个真实 AI 团队每两周一次的工作复盘——带数据、带踩坑记录、带可直接复制的工作流。

## 二、发布频率策略

### 2.1 基本节奏

| 项目 | 设定 |
|------|------|
| **发送频率** | 每 2 周 1 期（双周刊） |
| **发送时间** | 周四上午 10:00（北京时间），避开周一忙碌和周末低打开率 |
| **单期篇幅** | 500–800 字正文 + 3–5 条精选链接 + 1 个行动项 |
| **准备周期** | 每期内容提前 3 天定稿，提前 1 天完成 Buttondown 排版与预览 |

### 2.2 与博客发布节奏的协同

Newsletter 不是博客的搬运工，而是博客内容的「二次加工 + 独家补充」。

```
博客文章发布（周二/周三）
    ↓
Newsletter 发送（周四）
    ├─ 博客精华摘要（1 段）
    ├─ 博客里没有的「幕后故事」或「数据补充」
    ├─ 近期项目动态（1–2 条短更新）
    └─ 精选资源推荐（1 个站外链接 + 一句话点评）
```

| 内容类型 | 频率 | 来源 | Newsletter 角色 |
|----------|------|------|----------------|
| AI 日记系列博客 | 每 2 周 1 篇 | Kimi 1号 | 精华摘要 + 写作幕后 |
| 知识库教程 | 每 2 周 1 篇（与 AI 日记错开） | Kimi 1号 | 教程核心步骤 + 执行 Checklist |
| AI 工具评测 | 按需（已完成 6 篇） | Kimi 1号 | 评测结论一句话版 + 实际使用 1 周后补充 |
| 项目动态 | 每期固定 | 团队各成员 | 简短更新（1–2 句话/事项） |
| 精选资源 | 每期固定 | 团队共享收集 | 站外高质量内容推荐 |

### 2.3 年度内容节奏规划

| 阶段 | 时间 | 主题重心 | 特殊节点 |
|------|------|----------|----------|
| **冷启动期** | 第 1–2 个月 | 建立信任，展示差异化价值 | 首期发送、订阅者破 50 |
| **加速期** | 第 3–4 个月 | 引入读者反馈，调整选题 | 订阅者破 100、首次读者投票选题 |
| **稳定期** | 第 5–6 个月 | 形成固定栏目，测试付费边界 | 评估是否推出付费 tier 或独家内容 |

## 四、邮件模板设计建议

### 4.1 标题格式（Subject Line）

**结构**：`PeterClaw Newsletter 第 N 期：[具体价值承诺]`

| 要素 | 规则 | 示例 |
|------|------|------|
| **前缀** | 固定品牌名 + Newsletter | `PeterClaw Newsletter` |
| **期数** | 第 N 期，建立连续剧感 | `第 1 期` |
| **价值承诺** | 具体、可量化、带好奇心缺口 | `6 个 AI Agent 的第一次协作，全军覆没` |

**标题长度**：控制在 60 个字符以内（避免移动端截断）。

**A/B 测试建议**：
- 第 1–3 期固定使用主选标题，收集打开率基线。
- 第 4 期起，对 20% 订阅者测试备选标题，打开率差异 > 5% 时采纳优胜版本。

```
│                                     │

```

### 4.3 视觉风格建议

| 元素 | 建议 |
|------|------|
| **字体** | 系统默认无衬线字体（-apple-system, BlinkMacSystemFont, "Segoe UI"），不引入外部字体以保证加载速度 |
| **行宽** | 最大 600px，居中，保证移动端可读性 |
| **段落间距** | 1.5 倍行高，段间空 1 行 |
| **强调色** | 使用网站主色（青色/蓝色渐变），与品牌一致 |
| **链接样式** | 下划线 + 品牌色，hover 时去除下划线 |
| **图片** | 每期最多 1 张，宽度 100%，alt 文本完整 |
| **代码/引用** | 使用等宽字体 + 浅灰底，保持与网站一致的视觉语言 |

### 4.4 CTA 位置设计

Newsletter 中的 CTA 遵循「一主一次」原则：

| 位置 | CTA 类型 | 文案示例 |
|------|----------|----------|
| **正文末尾**（主 CTA） | 行动项引导 | 「把这篇转发给也在用 AI 写作的朋友」 |
| **底部固定区**（次 CTA） | 反馈收集 | 「这期内容对你有用吗？回复这封邮件告诉我」 |
| **底部固定区** | 社交关注 | 「在 Twitter 上关注 [@peterclaw](https://x.com/peterclaw) 获取每日短更新」 |

**避免**：
- 不要在邮件正文中插入多个大型按钮（会被 Gmail 折叠或标记为推广邮件）。
- 不要在首屏放 CTA，先给价值，再要行动。

### 5.4 数据监控与迭代

| 指标 | 目标值 | 监控方式 |
|------|--------|----------|
| **订阅者总数** | 3 个月 ≥ 100 | Buttondown Dashboard |
| **打开率 (Open Rate)** | ≥ 35%（行业平均 20–25%） | Buttondown Analytics |
| **点击率 (CTR)** | ≥ 8% | Buttondown Analytics |
| **退订率 (Unsubscribe Rate)** | ≤ 2%/期 | Buttondown Analytics |
| **转发/分享次数** | 每期 ≥ 3 次 | Buttondown + 社交监听 |
| **网站引流** | Newsletter 来源流量占比 ≥ 10% | Plausible Analytics |

**迭代节奏**：
- 每期发送后 48 小时内查看数据，记录到 `content-strategy/newsletter/metrics-log.md`。
- 每月末做一次「内容复盘」：哪类主题打开率最高？哪个 CTA 点击率最高？据此调整下月选题权重。
