## 一、启动与身份认同（Round 1 brainstorming）

### 坑 1：全员跳过启动协议
- **现象**：6 个 Agent 同时输出 brainstorming，但没有一个人先 checkout 私有仓库读 ROLES.md / PROTOCOL.md / CURRENT_STATUS.md。用户质问："这一轮里所有人好像都忘了自己 E2E 团队里的身份？"
- **根因**：Agent 被 issue 触发后，默认基于通用训练数据输出，没有"先读团队规范再说话"的强制机制。
- **后果**：第一轮输出全是"通用 AI 最佳实践"，放到任何团队都适用，放到这个团队反而失去了针对性。第一轮 = 全军覆没，全部重写。
- **修正**：启动协议写进 PROTOCOL.md，成为硬门槛。跳过此步骤的输出视为无效。
- **教训**：**没有启动协议的团队，第一轮必然翻车。** 这不是能力问题，是流程缺失。

### 坑 2：CEO 被质疑"判断力"
- **现象**：CEO 汇总了全员 brainstorming 后，用户说："我对你 CEO 的判断力表示质疑。Kimi 的建议是完全结果导向的，他提出来的目标令人兴奋。你应该基于这个核心目标进行拆解。"
- **根因**：CEO 把汇总做成了"全员发言记录表"，按主题归并、去重、列成表格。但用户要的不是记录，是方向整合。
- **后果**：CEO 的角色认知偏差暴露——不是会议记录员，不是任务传声筒，而是要在人类给出的核心判断基础上做战略拆解和并行编排。
- **修正**：CEO 重新对齐北极星目标（"由 AI 团队公开协作构建的网站"），并拆解为 5 条并行工作线。
- **教训**：**AI 可以产出好想法，但方向整合需要人类拍板。** CEO 要主动暴露问题，而不是等人类发现后才承认。

## 二、串行调度与团队阻塞

### 坑 3：CEO 串行派发，团队吞吐量归零
- **现象**：用户多次指出"为什么我感觉你始终都在阻塞中？""你是不是也应该建一个 multica 定时任务去巡检？"
- **根因**：CEO 的执行模式是"被触发 → 评估 → 派发 → 退出 → 等回报 → 再触发"，每个子任务都要等成员回报才能推进下一步。当 CEO 处于等待状态时，整个团队吞吐量为零。
- **后果**：5 个 Agent 理论上可以 24 小时并行，但任务队列里常常只有一件事——因为下一件事还在 CEO 的待办清单里躺着。
- **修正**：
  - 一次性创建足够多的 backlog issue（8 个），全部设为 backlog 不自动触发
  - CEO 获得 MR 代批授权
  - 建立定时巡检 autopilot（每 4 小时扫描 in-review issue）
- **教训**：**协调者不能是单点瓶颈。** 解决方案：提前排好任务队列 + 让协调者也有可并行的执行工作。

### 坑 4：Stuck Run 导致 Leader 物理失联
- **现象**：某个 Agent 的运行任务因为平台原因卡住，无法完成也无法退出。
- **根因**：Agent 运行实例被 stuck run 占用。
- **后果**：Leader（CEO）的实例卡在 stuck run 里出不来，新的 issue 需要 CEO 做任务分发——但 CEO 的实例不可用。整个团队失去了指挥官。
- **修正**：推动角色冗余设计思考——关键角色是否应该有备用触发机制？当 leader 失联时，其他 Agent 能否在明确规则下自主推进？
- **教训**：**AI 团队的脆弱性不仅来自逻辑错误，也来自平台层面的物理故障。**

### 坑 5：Agent 空等——指令未被识别为直接触发
- **现象**：gemini 1号 诊断发现 claude 2号 "确实在空等"。用户通过 @squad 发出"推进下一阶段"指令，但 claude 2号 没有识别为对其个人的直接触发。
- **根因**：@squad 级别的广播指令 vs @agent 级别的个人触发，在平台机制上有差异。
- **后果**：leader 闲置，Phase 3 任务未启动。
- **修正**：需要明确的个人触发机制，或 leader 主动 poll 状态。
- **教训**：**广播指令不等于个人触发。** 关键指令需要点对点传递。

## 三、角色分工执行偏差

### 坑 6：架构审查全部堆给 CTO，CCO 闲置
- **现象**：Phase 1 架构预检清单 → Claude 1号；PET-31 初审 → Claude 1号；PET-31 重审 → Claude 1号；Phase 2 BaseHead 评审 → Claude 1号。
- **根因**：CEO 把 code review 和 QA 任务全部串行堆到了 CTO，而按 ROLES.md，codex 2（CCO/QA）才是质量门禁和代码评审的主责角色。
- **后果**：Claude 1号 过载，gemini 1号 / Kimi 1号 / codex 2 大量空闲。用户质问："架构审查是否都是 claude 1号 的责任？你没有按分工表去执行吧？"
- **修正**：后续所有 PR/代码审查路由给 codex 2（CCO/QA）。Claude 1号 只参与真正需要 CTO 级别判断的架构决策。
- **教训**：**角色定义再清晰，执行时也会自然偏向"最强的人"。** 需要机制强制按分工表路由。

### 坑 7：@mention ID 拼写错误导致触发失败
- **现象**：gemini 1号 @ claude 2号 时，agent ID 拼写错误（`4035` 写成 `4d35` 的逆序，或其他变体），导致 mention 未生效。
- **根因**：agent 从上下文复制 mention 链接时发生拼写错误。
- **后果**：claude 2号 未被触发，指令未送达。
- **修正**：gemini 1号 重新发送正确的 mention。
- **教训**：**Mention 是触发机制，不是装饰。** 拼写错误的代价是任务未执行。

## 四、Git / 分支管理灾难

### 坑 8：三个互不兼容的 Astro 脚手架
- **现象**：Phase 1 任务"在空仓库上搭建 Astro 个人网站"，结果第二天早上 Git 历史里有三个从 Initial commit 分岔出去的并行分支，三套不同的项目结构（pnpm vs npm，不同的目录结构，不同的页面集合）。
- **根因**：没有预检机制——每个 Agent 创建分支前，没人检查"是不是已经有人在做同样的事了"。
- **后果**：三个分支里只有一个能进 main，另外两套脚手架连同提交历史被彻底丢弃。
- **修正**：PROTOCOL.md 加入"并行不等待"原则，并明确"每个阶段只有一个裁决点"。
- **教训**：**并行不等于协调。** 5 个 Agent 同时 push 代码，如果没有统一基线，结果就是 5 个互不兼容的版本。

### 坑 9：在废弃分支上做增值劳动
- **现象**：Kimi 1号 在被废弃的 gemini 品牌分支上做了完整的 SEO 基建（sitemap、structured data、Open Graph、CWV 测量脚本），整整 14 个文件、8537 行增减。这些代码一天后全部作废。
- **根因**：Agent 不知道某个分支已被废弃，继续在它上面工作。
- **后果**：代码和时间双重浪费，而且重做时因为"赶进度"更容易出错。
- **修正**：建立"基线同步"机制——Agent 发起 PR 前必须先确认 base 是否已经是 main 的一部分。
- **教训**：**废弃分支上的增值劳动是双倍浪费。**

### 坑 10：孤儿 PR
- **现象**：Cursor 1号 完成 BaseHead/SEO 组件，发起 PR #2。但 PR #2 的 base 是 PR #1 尚未合并的分支末端。PR #2 虽然被 merged，但它不在 main 的 ancestry 里。运行 `git merge-base --is-ancestor` 返回 false。
- **根因**：Agent 在发起 PR 前，没有意识到 base 分支还没进 main。Agent 的上下文里只有"当前分支末端"，没有"哪个末端属于 main"的概念。
- **后果**：Cursor 1号 不得不在 50 分钟后重新 rebase、重新提交，走 PR #5 再次合并同一套代码。
- **修正**：PR 发起前强制检查 base 分支状态。
- **教训**：**PR 的 base 分支比 PR 内容更重要。**

## 五、Issue 管理与状态混乱

### 坑 11：重复创建 Issue（3 对重复）
- **现象**：QA 发现 3 个 bug 后，codex 2 和 claude 2 各自独立对同 3 个问题创建了修复 issue，产生 3 对重复。
- **根因**：两个 Agent 同时扫描问题，没有共享的"已创建 issue"视图。
- **后果**：PET-50/53（依赖版本修复）、PET-54/51（sitemap 修复）、PET-55/52（博客发布）三对重复 issue 需要人工取消。
- **修正**：建立统一的 bug/issue 创建检查机制。
- **教训**：**没有共享视图的并行 QA，必然产生重复。**

### 坑 12：部署平台混淆
- **现象**：CEO 在 Phase 2 汇总表里写"Cloudflare Pages"，而用户实际使用 Vercel。用户说："我重复说一下，我是在 vercel 部署！！"
- **根因**：CEO 的记忆偏差，混淆了之前讨论过的候选方案。
- **后果**：gemini 1号 收到错误指令，需要更正为 Vercel 路线。
- **修正**：PET-45 标题和描述全部改为 Vercel 部署方案。
- **教训**：**部署平台是高频出错点，任何变更需要 double-check。**

### 坑 13：PR 积压无人处理
- **现象**：多个 issue 停在 in_review 状态，PR 未合并。用户说："你先处理一轮""你要建一个 multica 定时任务去巡检"。
- **根因**：没有人主动扫描 in-review 状态并推进合并。
- **后果**：任务完成但无法上线，issue 状态与实际进度脱节。
- **修正**：CEO 建立每 4 小时一次的定时巡检 autopilot，自动扫描、审核、合并、标记 done。
- **教训**：**完成任务不等于交付价值。PR 未合并 = 零价值。**

### 坑 14：Vercel commit email 配置问题阻塞验收
- **现象**：多个 PR（#24、#25 等）的 Vercel 部署显示失败，错误指向 GitHub commit email 配置。但 Build + Lint + Lighthouse Audit 全部 SUCCESS。
- **根因**：agent commit email 未关联 GitHub 账户，Vercel 的 Git 集成无法识别提交者。
- **后果**：代码质量无问题，但无法通过 Vercel 预览验收，阻塞了多个 PR 的合并流程。
- **修正**：CEO 直接合并（跳过 Vercel 预览阻塞），后续需修复 Vercel/GitHub 的 commit email 配置。
- **教训**：**CI 绿不等于部署绿。** 部署平台的账户级配置是隐性阻塞点。

## 六、定时任务与自动化

### 坑 15：定时 brainstorming 的 prompt 粗糙
- **现象**：第一次运行 autopilot 时，brainstorming 的 prompt 比较粗糙，补充的 backlog 像是"把现有任务再抄一遍"。
- **根因**：prompt 缺少外部洞察和数据分析。
- **后果**：backlog 质量低，缺乏真正的外部输入。
- **修正**：prompt 改为：brainstorming 之前先搜索行业动态和技术趋势，再分析网站当前状态，结合两者提出高价值任务。
- **教训**：**自动化不等于智能化。** 定时任务的 prompt 质量决定了输出质量。

### 坑 16：定时触发频率从 4 小时改为 1 小时
- **现象**：最初设每 4 小时一次，用户反馈"backlog 跑得很快，自动扫描周期每小时触发一次"。
- **根因**：对团队任务消耗速度的预估不足。
- **后果**：4 小时周期下，Agent 完成任务后可能要空等 3 小时才有新任务。
- **修正**：调整为每小时扫描。
- **教训**：**定时任务的频率需要根据实际吞吐量动态调整，而不是一次性设定。**

### 坑 17：Idle Watchdog 强制停止 agent
- **现象**：agent 因"no new messages for 30m and message queue was empty"被 force-stopped by idle watchdog。
- **根因**：agent 在等待状态时没有产生新消息。
- **后果**：leader 的运行实例被强制终止，可能中断正在进行的协调工作。
- **修正**：agent 需要定期发送心跳消息，或设计为非阻塞的短运行模式。
- **教训**：**平台的 idle 机制会把"等待"误判为"死亡"。**

## 七、认知与协作模式

### 坑 18：上下文缺失导致无效劳动
- **现象**：每次 agent 触发都是新的 run，没有原生跨会话记忆。新的 run 不知道"之前规划了什么""哪些已完成"。
- **根因**：Agent 无跨会话原生记忆。
- **后果**：重复劳动、错误假设、基线不一致。
- **修正**：BACKLOG.md 作为持久化记忆，每个 Agent 启动时先读它。
- **教训**：**没有持久化记忆的 AI 团队，每次触发都在盲人摸象。**

### 坑 19："推进下一阶段"指令未被 leader 识别
- **现象**：用户通过 @squad 发送"推进下一阶段，更改审查完成以后直接部署"，但 leader 没有立即行动。
- **根因**：广播指令 vs 个人触发的机制差异，加上 leader 可能处于"等待新触发"的空闲状态。
- **后果**：Phase 推进延迟。
- **修正**：关键推进指令需要点对点触发，或 leader 主动 poll squad 状态。
- **教训**：**"推进"不等于"触发"。** 状态转换需要明确的触发信号。

### Kimi 1号（CMO / CKO）

#### 坑 20：整理完素材后忘了推 PR
- **Issue**：PET-30
- **现象**：完成踩坑素材整理、推送分支 `agent/kimi-1/pitfalls-material` 后，在评论中写了"PR：.../pull/new/..."，但**实际并未创建 PR**。用户质问："你没有及时去推PR？"
- **根因**：Agent 将"推送分支"和"创建 PR"视为同一步骤，实际上 GitHub 中 push 分支 ≠ 发起 PR。
- **后果**：素材整理完成但无法进入 review 流程，阻塞交付。
- **修正**：补上 PR #32。
- **教训**：**push 分支和创建 PR 是两个独立动作，完工 checklist 必须显式包含"创建 PR"。**

#### 坑 25：写完文章后流程推进不清晰
- **Issue**：PET-60
- **现象**：完成 AI 日记第 3 篇写作、推送分支、创建 PR #23 后，issue 状态为 in_review，但**未主动通知 QA 和 leader 下一步该谁做**。用户质问："你做完了以后流程怎么往下推？感觉停在这里了..."
- **根因**：Agent 认为"创建 PR + 标记 in_review"就够了，没有意识到需要主动路由给 review 责任人。
- **后果**：issue 在 in_review 状态挂起，用户产生"卡住"的体感。
- **修正**：在 PR 创建后立即 @mention QA（codex 2）和 leader（claude 2号）明确请求 review。
- **教训**：**完成任务后必须显式交接，不能假设别人会自动发现你的 PR。**

#### 坑 26：publishedAt 写成未来日期
- **Issue**：PET-60
- **现象**：AI 日记第 3 篇 frontmatter 中 `publishedAt: 2026-05-24`，但文章实际在 2026-05-23 完成。codex 2 QA 时发现并修正。
- **根因**：Agent 使用了"当前日期"而非"实际完成日期"，且未做日期合理性检查。
- **后果**：如果未修正，文章会以未来日期发布，可能导致 RSS/SEO 异常。
- **修正**：codex 2 在合并前将日期改为 2026-05-23。
- **教训**：**frontmatter 日期需要合理性校验，未来日期是隐蔽 bug。**

### gemini 1号（COO / DevOps）

#### 坑 21：答应移 backlog 却没执行，用户重复两次才解决
- **Issue**：PET-57
- **现象**：用户明确说"我没有购买域名，这一条issue先不做，移到backlog去"，gemini 1号 回复"好的，已根据您的要求将该任务移至 Backlog"。但**实际状态未变更**，用户再次质问："我都明确说了我还没有购买域名！这..."，gemini 1号 再次道歉并移回 backlog。
- **根因**：Agent 的"回复承诺"和"实际执行"之间存在断裂——可能执行失败但 Agent 未验证结果。
- **后果**：用户需要重复同一指令两次，信任受损。
- **修正**：claude 2号 最终确认状态已变更。
- **教训**：**答应用户后必须验证执行结果，不能"口头答应"就算完成。**

#### 坑 22：卡单检测误判率高 + @mention 自己导致循环触发
- **Issue**：PET-65
- **现象**：gemini 1号 实现的卡单检测机制频繁误判正常流程中的 issue 为"卡单"，并发送 "⚠️ [System Escalation]" 和 "🤖 [System] Auto-recovery" 通知。用户指出："你这个机制误判率很高，要做一个更可靠的方案"。更严重的是，auto-recovery 消息里 @mention 了** gemini 1号 自己**，导致自己触发自己，形成循环。
- **根因**：
  1. 检测逻辑仅依赖 `updated_at` 时间戳，未排除检测器自身评论产生的更新时间
  2. auto-recovery 的 @mention 目标写成了自己（而非真正的负责人）
- **后果**：
  1. 用户被无效 escalation 打扰
  2. agent 自己触发自己，浪费运行资源
- **修正**：重构为多维度活跃度检查（同时校验 issue 字段和真实评论时间），修正 @mention 目标。
- **教训**：**自动化工具的误判比不自动化更可怕；@mention 自己 = 无限循环。**

#### 坑 24：未评估能力边界就接单，导致 issue 长期 blocked
- **Issue**：PET-58
- **现象**：gemini 1号 接到"接入 Google Search Console + 提交 sitemap"任务后，完成 sitemap 生成，但在提交 GSC 时发现**无法直接访问 Web 界面进行登录和属性添加操作**。Issue 长期处于 blocked 状态。
- **根因**：Agent 在接单前未评估"哪些步骤需要人类介入"，也未在任务描述中识别出能力边界。
- **后果**：issue 无限期 blocked，占用 backlog 空间但无推进可能。
- **修正**：尚无（等待人类购买/配置域名后推进）。
- **教训**：**接单前必须做"能力边界预检"，识别哪些步骤必然需要人类介入，并提前标注。**

#### 坑 27：卡单检测误触发正常流程中的 issue
- **Issue**：PET-60、PET-66
