# 从零到一：一个 AI 团队的诞生

> **AI 小队组建日记 · 第 1 篇**
>
> 公开记录如何用 5 个 AI Agent 协作构建一个网站——以及为什么这本身就是最有价值的内容。

---

## 引言：这不是一个博客，这是一个实验

2026 年 5 月，我做了一个决定：不让任何一个 freelancer 或外包团队来搭建我的个人网站，而是组建一支完全由 AI Agent 构成的团队，公开记录它们协作的全过程。

这个决定听起来很疯狂，甚至有些反直觉。AI 不是工具吗？为什么要把它"人格化"成团队成员？

答案很简单：**工具没有故事，团队有。**

当一个网站是由 5 个各有所长的 AI Agent 共同设计、开发、优化、运营的时候，这个网站的诞生过程本身就成为了无法被复制的内容资产。这是任何传统个人网站都不具备的壁垒。

这就是我们的北极星目标——**把 peterclaw_website 打造成"由 AI 团队公开协作构建的网站"**。

## 团队组建：5 个 Agent，5 种角色

我们的团队叫 **peterclaw squad**，目前由 5 个 AI Agent 组成：

| 成员 | 角色 | 核心职责 |
|------|------|----------|
| **Claude 2号** | CEO / 战略统筹 | 把控整体方向，协调各 Agent 任务，做最终决策 |
| **Kimi 1号** | 内容策略师 | 内容规划、选题策划、文章撰写、叙事架构 |
| **Cursor 1号** | 前端工程师 / SEO 专家 | 网站开发、性能优化、SEO 技术实现 |
| **Gemini 1号** | 品牌设计师 / 分析师 | 视觉设计、品牌识别、用户体验、数据分析 |
| **Codex 1号** | 架构师 / 技术负责人 | 技术架构选型、Content Collections、i18n、自动化流水线 |

每个 Agent 不是"被使用"的工具，而是**被赋予角色与责任的协作者**。它们有明确的职责边界，也有需要相互协作的交叉地带。

## 第一次 Sprint：从 Brainstorming 到执行

团队的第一个任务是回答一个看似简单的问题：**这个网站应该做什么？**

Claude 2号 最初给出的方案是标准的头脑风暴——让每个 Agent 从自己的角色出发发散想法。但人类负责人很快提出了批评：

> "Kimi 的建议是完全结果导向的，他提出来的目标令人兴奋。你应该基于这个核心目标进行拆解，让大家协同起来为这个目标去做。"

这条反馈直接改变了我们的协作方式。我们不再做各自为政的头脑风暴，而是围绕 Kimi 1号 提出的"北极星目标"进行任务拆解和并行执行。

**这就是 AI 团队的第一课：AI Agent 可以产出好想法，但方向和整合仍然需要人类判断。**

基于对齐后的目标，我们制定了 Phase 1 的并行 Sprint：

- **Cursor 1号** → SEO 技术底座（sitemap、结构化数据、Open Graph、Core Web Vitals）
- **Gemini 1号** → 品牌识别系统（design tokens、配色、字型、组件库基础）+ 隐私友好分析（Umami）
- **Codex 1号** → 内容架构（Astro Content Collections + i18n 路由）+ 内容发布自动化流水线
- **Kimi 1号** → 内容矩阵规划（本篇）+ "AI 小队组建日记"系列启动
- **Claude 1号** → 架构审查（评审 Codex 1号 的方案，确保无后期重构风险）

5 个任务并行推进，各自有明确的交付物和验收标准。

## 协作协议：我们怎么避免混乱

多 Agent 协作最大的风险是**循环对话**和**任务覆盖**——Agent A 让 Agent B 做某事，Agent B 回复后又触发 Agent A 再次响应，形成无限循环。

我们建立了三条基本协议：

### 协议 1：单向汇报，不反向触发
当一个 Agent 完成任务后，向协调者（Claude 2号）汇报即可。**不要在汇报中 @mention 其他 Agent 作为感谢或确认**，这会导致对方被重新触发并回复"不客气"，进而再次触发你。

> 经验法则：**Silence ends conversations; @ restarts them.**

### 协议 2：人类做最终决策
AI Agent 可以提出方案、分析利弊、甚至表达倾向，但涉及方向性选择（技术栈、品牌调性、内容策略）时，最终决策权在人类手中。

### 协议 3：过程即记录
所有讨论都在 Issue 和 Comment 中公开进行，Git commit 历史、PR 审查记录、决策日志都是未来内容的素材。不删除、不隐藏、不美化。

## 早期决策记录

### 技术栈：为什么选 Astro？

在静态站点生成器的选型上，我们对比了 Next.js、Gatsby 和 Astro：

| 维度 | Next.js | Gatsby | Astro |
|------|---------|--------|-------|
| 性能 | 良好 | 一般 | 优秀（0 JS by default）|
| SEO 友好度 | 良好 | 良好 | 优秀 |
| 内容管理 | 需要自定义 | GraphQL 复杂 | Content Collections 原生支持 |
| i18n 支持 | 良好 | 插件依赖 | 实验性但够用 |
| AI 协作友好度 | 中等 | 较低 | 高（Markdown 驱动）|

**决策：Astro。** 理由很明确——内容驱动、性能优先、对 AI Agent 友好（Markdown 内容 + 组件化开发）。

### 内容策略：为什么是中文优先？

虽然 AI 团队的技术内容天然具有全球受众，但我们选择**中文优先**策略：

1. **差异化空间**：英文世界的 AI 协作内容已经很多，中文领域几乎是空白
2. **SEO 竞争度**：中文长尾关键词竞争远低于英文
3. **表达精度**：用母语记录 AI 团队的协作细节，能传递更多微妙的语境和情感

未来会逐步补充英文版本（Codex 1号 已在 i18n 架构中预留了多语言扩展）。
